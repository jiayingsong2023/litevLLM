# LitevLLM 90天改造计划（按周拆解 + 人力估算 + 风险）

> 目标：在不破坏当前“单卡优先 + 性能优先”优势前提下，补齐工程治理短板，建立可持续交付闭环。
> 执行原则：**文档真实一致、测试入口统一、依赖分层、观测先行、风险前置**。

---

## 1. 计划范围与成功标准

### 1.1 范围（90天 / 13周）
- 文档与代码一致性治理（Capability Matrix、stub治理、命令一致性）
- CI与测试入口重构（smoke/契约/回归三层）
- 依赖分层（core/serving/multimodal/dev）与 `uv` 工作流统一
- 观测与性能回归（延迟、吞吐、OOM/重试、回归报告）
- 发布治理（发布门禁 + 回归摘要 + 能力快照）

### 1.2 成功标准（建议量化）
- 文档命令可执行率 ≥ 95%
- 关键路径测试（smoke + 契约）PR 阶段通过率 ≥ 98%
- 回归发现问题到定位时间（MTTD）下降 30%
- 依赖安装时长下降 25%（core profile）
- 发布包附带能力矩阵快照与回归摘要覆盖率 100%

---

## 2. 13周分周执行（基础版）

> 该分周是“标准节奏模板”，后续 1人/3人/5人版本会在此基础上做并行度调整。

### 第1周：基线盘点与冻结
- 任务
  - 盘点 docs/CI/tests 三方命令入口差异
  - 建立“当前能力状态清单”（完整/实验/兼容壳/废弃）
  - 设立改造分支策略与里程碑看板
- 交付物
  - Gap List（文档-代码-CI差异清单）
  - 里程碑燃尽图（13周）
- 风险
  - 范围蔓延（一次想改完所有模块）
- 缓解
  - 采用“核心路径优先”（推理入口、常用模型、API主路径）

### 第2周：Capability Matrix v1
- 任务
  - 新建单一事实源文档（建议 `docs/CAPABILITY_MATRIX.md`）
  - README/API/架构文档统一引用该矩阵
- 交付物
  - Capability Matrix v1（含状态定义、模块映射、负责人）
- 风险
  - 团队对状态标签理解不一致
- 缓解
  - 在矩阵开头给出判定规则与示例

### 第3周：Stub治理规范
- 任务
  - 统一 stub 标识策略（注释协议/显式异常）
  - 区分“兼容壳（可用）”与“占位（不可用）”
- 交付物
  - Stub Policy（代码注释模板 + 评审检查项）
- 风险
  - 误把兼容层当缺陷删除
- 缓解
  - 为分布式 shim 等保留层建立白名单

### 第4周：测试入口统一（Smoke层）
- 任务
  - 定义单一 smoke 入口（脚本/pytest 目录二选一，建议保留脚本并在pytest包装）
  - CI只调用统一入口，避免多套命令分叉
- 交付物
  - `smoke.yml` 与 tests README 对齐
- 风险
  - 历史脚本参数差异导致回归遗漏
- 缓解
  - 先做“旧入口映射表”再切换

### 第5周：契约测试（API/错误码/参数校验）
- 任务
  - 建立API schema快照测试
  - 增加错误码一致性与异常路径测试
- 交付物
  - 契约测试集v1（覆盖高频接口）
- 风险
  - 契约冻结过早影响迭代
- 缓解
  - 设置“兼容窗口”和变更公告机制

### 第6周：并发与取消语义测试
- 任务
  - 补充并发提交、超时、取消、重试语义测试
  - 记录边界行为（队列长度、拒绝策略）
- 交付物
  - 并发语义测试集v1
- 风险
  - 测试不稳定（flaky）
- 缓解
  - 固化随机种子 + 分层超时 + 重试阈值

### 第7周：依赖分层设计
- 任务
  - 拆分 `pyproject` 依赖 profile：core / serving / multimodal / dev-test
  - 设计 `uv sync --extra ...` 推荐矩阵
- 交付物
  - 依赖分层 RFC + 安装时长对比基线
- 风险
  - extras组合爆炸
- 缓解
  - 官方只维护少量“黄金组合”

### 第8周：依赖分层落地 + `uv` 标准化
- 任务
  - 改造 README、开发文档、CI 使用 `uv run` / `uv sync`
  - 建立最小可运行环境（core）
- 交付物
  - 统一命令手册（开发/测试/发布）
- 风险
  - 本地缓存差异导致“我这可跑你那不行”
- 缓解
  - 锁定 lockfile，CI强制冷启动验证

### 第9周：观测指标最小闭环
- 任务
  - 打通请求级 p50/p95 延迟、token吞吐、OOM次数、重试次数
  - 输出统一日志字段（request_id/model/profile）
- 交付物
  - 指标字典 + 首版dashboard（可简版）
- 风险
  - 指标定义不一致导致对比失真
- 缓解
  - 指标字典先行，评审后再埋点

### 第10周：性能回归自动摘要
- 任务
  - 建立性能基线快照与阈值告警
  - 自动生成回归摘要（模型维度/硬件维度）
- 交付物
  - perf regression report v1
- 风险
  - 硬件漂移影响可比性
- 缓解
  - 固定“标准机型 + 标准驱动版本”

### 第11周：发布门禁（Release Gate）
- 任务
  - 发布流程强制附带：能力矩阵快照 + 回归摘要 + 已知限制
  - 增加文档命令存在性检查（路径必须存在）
- 交付物
  - 发布检查清单 v1
- 风险
  - 门禁过严拖慢发版
- 缓解
  - 分“必过项/可豁免项”，豁免需记录

### 第12周：试运行与纠偏
- 任务
  - 选两次真实需求做端到端试运行
  - 统计失败原因与流程耗时
- 交付物
  - 试运行复盘报告
- 风险
  - 试运行样本偏少
- 缓解
  - 覆盖至少一个新模型 + 一个回归修复场景

### 第13周：固化与移交
- 任务
  - 输出SOP（开发/测试/发布）
  - 完成角色交接与培训
- 交付物
  - 90天总结与下一季度Backlog
- 风险
  - 改造成果未制度化，后续回退
- 缓解
  - 将关键检查项固化为CI规则

## 3. 人力版本对照（1人 / 3人 / 5人）

## 3.1 单人版（1人）

### 适用场景
- 维护型团队，需求波动较大，无法并行推进多个流。

### 资源与节奏
- 建议投入：1.0 FTE（每周 30~40 小时）
- 交付策略：**串行推进，保证每两周一个可见里程碑**

### 周计划压缩建议
- W1-W2：基线盘点 + Capability Matrix
- W3-W4：Stub治理 + Smoke统一
- W5-W6：契约测试（仅高频接口）
- W7-W8：依赖分层（先core+dev）
- W9-W10：最小观测闭环
- W11-W12：发布门禁最小集
- W13：文档固化与复盘

### 风险（单人版高发）
- Bus factor = 1，关键人不可用即停摆
- 容易被线上事务打断，里程碑滑动

### 缓解
- 每周固定半天“改造保护时间”
- 每个里程碑必须有“可回滚点”
- 所有决策记录在 ADR，避免隐性知识

---

## 3.2 小队版（3人）

### 角色建议
- A（架构/治理）：文档一致性、发布门禁、流程规则
- B（测试/CI）：smoke、契约、并发语义、CI落地
- C（平台/性能）：依赖分层、观测埋点、性能回归

### 资源与节奏
- 建议投入：2.4~2.8 FTE（考虑20%维护噪音）
- 交付策略：**双线并行（治理线 + 质量线）**

### 周计划并行建议
- W1-W2：A做矩阵，B做入口梳理，C做依赖基线
- W3-W4：A推进stub规范，B完成smoke统一，C出分层RFC
- W5-W6：B做契约+并发测试，A做文档校验规则，C推进uv落地
- W7-W9：C主导观测闭环，B补测试稳定性，A固化发布清单
- W10-W13：联调、试运行、SOP与培训

### 风险
- 多人并行导致规范分叉（命名、目录、脚本）
- CI时长膨胀影响开发效率

### 缓解
- 每周一次30分钟“规范同步会”
- CI采用分层触发（PR快速层 / 夜间完整层）

---

## 3.3 强化版（5人）

### 角色建议
- TL/PM（1）：排期、风险与跨模块协调
- 质量工程（2）：测试金字塔、CI稳定性、发布门禁
- 平台工程（1）：依赖分层、工具链、开发体验
- 性能工程（1）：观测、性能基线、回归分析

### 资源与节奏
- 建议投入：4.0~4.5 FTE
- 交付策略：**三线并行（治理线 + 质量线 + 性能线）**

### 周计划加速建议
- W1即启动三线并行，W4完成P0（真实性修复）
- W8完成P1主体（分层依赖+中层测试）
- W10完成观测+回归自动摘要
- W11-W13以发布门禁试运行为主

### 风险
- 组织成本上升（沟通、评审、冲突）
- 过度工程化，影响业务需求响应

### 缓解
- 采用“轻流程 + 强自动化”
- 每两周做价值回顾，砍掉低收益改造项

## 5. 风险雷达（建议持续跟踪）

- 技术风险
  - 兼容壳与真实能力边界不清，导致误用
  - 测试不稳定影响信任度
- 进度风险
  - 线上需求插队打断改造节奏
- 组织风险
  - 责任边界不清导致“都在做、没人闭环”
- 外部风险
  - 驱动/硬件/上游模型变更引发性能噪声

### 风险治理机制
- 每周固定更新风险清单（概率/影响/责任人/缓解动作）
- 红黄绿分级：红色项必须在当周复盘
- 关键门禁失败必须有“恢复时间承诺”（ETA）

## 6. 建议的立即行动（本周可执行）

1. 建立 `docs/CAPABILITY_MATRIX.md`（先覆盖核心模块）
2. 统一并公布唯一 smoke 入口，更新 CI 与 tests 文档
3. 把开发/测试命令全部切换到 `uv run` 示例
4. 选 3 个高频 API 补契约测试（先求有，再完善）
5. 确定 1/3/5 人版本中的一个作为当前季度执行基线

---

## 7. 选择建议（如何在三种版本中决策）

- 如果你们当前只有 1 位核心维护者：选 **1人版**，目标是“稳态可持续”，不要追求全覆盖。
- 如果你们有 2~3 位稳定贡献者：选 **3人版**，性价比最高，能兼顾治理与交付。
- 如果你们正准备对外发版或承接更多模型：选 **5人版**，优先打通发布可信度与性能可观测。
